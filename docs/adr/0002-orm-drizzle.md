# 0002. Use Drizzle as the ORM

## Status

Accepted, 2026-05-15.

## Context

Vellum's data layer is dominated by ledger queries: pulling journal entries with their lines, aggregating balances per account, computing time-windowed reports across thousands of entries. These queries are JOIN-heavy and benefit from precise SQL control.

The schema also needs Postgres-specific features:

- CHECK constraints on journal entries to enforce the double-entry invariant at the database level (sum of debits equals sum of credits per entry). This is the last line of defense if the application layer ever gets it wrong.
- Partial indexes on `idempotency_keys(expires_at)` for cheap cleanup.
- Generated columns for normalized amounts in a reporting currency.

TypeScript integration must be tight. We do not want a separate codegen step that drifts out of sync between local dev, CI, and production. Migrations must be plain files in the repo, runnable by `pnpm` scripts in CI and on a deploy host.

## Decision

Use Drizzle as the primary ORM for `apps/api`.

- Schema lives in `apps/api/src/db/schema/`, one file per aggregate.
- Migrations live in `apps/api/src/db/migrations/`, generated by `drizzle-kit generate` and committed to the repo.
- Raw SQL is allowed and expected for ledger aggregation queries; we use `drizzle-orm/sql` template tags so column types remain inferred.
- The journal entry repository wraps query construction so the rest of the code does not import the Drizzle client directly.

## Options considered

**Prisma.** Strongest DX in the space and owns the migration story end-to-end. The query engine is opaque, complex JOINs produce N+1 patterns unless you drop to `$queryRaw`, and the relational connector that improved this still ships an engine binary and a codegen step we do not want in a self-hostable app. Rejected.

**Drizzle.** Schema is TypeScript, types flow from schema, generated SQL is predictable, and the raw SQL escape hatch is first-class rather than an afterthought. Migrations via `drizzle-kit` are plain SQL files that diff cleanly in PRs. Smaller community than Prisma and Drizzle Studio is younger. Picked.

**Kysely.** Type-safe query builder, not an ORM. Considered seriously. Rejected for v1 because we would be assembling four moving parts (kysely + kysely-codegen + a migration runner + relation loading) that Drizzle bundles. Kept on the shortlist if we ever outgrow Drizzle.

**TypeORM.** Active Record pattern, decorator-heavy, weaker type inference than the options above. Rejected.

**Raw SQL with Slonik.** Maximum control, weakest ergonomics for the 80% of queries that are not ledger aggregations. Saved as the escape route, not the default.

## Consequences

Single source of truth for schema and TypeScript types; no codegen drift. We can write the ledger CHECK constraints directly in migration SQL without fighting an ORM that assumes simpler invariants. Drizzle Studio replaces ad-hoc DB inspection; if we miss something later we add DBeaver alongside it.

We accept a smaller ecosystem than Prisma: fewer Stack Overflow answers, occasional sharp edges in `drizzle-kit` around complex schema changes, fewer third-party plugins. Worth it for the control over generated SQL.

If Drizzle becomes unmaintained, migration to Kysely is mechanical because the schema is already SQL-first and our repositories already isolate query construction. The cost of a hypothetical swap is bounded.
