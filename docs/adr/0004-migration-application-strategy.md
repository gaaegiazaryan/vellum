# 0004. Migration application strategy

## Status

Accepted, 2026-05-16.

## Context

ADR-0002 picked Drizzle as the ORM and committed to migration files generated by `drizzle-kit` and stored in `apps/api/src/db/migrations/`. That ADR did not say _who runs them, when, or how a failed migration is recovered_. Those questions come due the moment the first migration lands.

Three operational concerns frame the decision:

- A migration that runs during request-serving traffic is dangerous. Long lock contention on a busy table, or a migration that fails halfway, leaves the app in a state where some pods are on the new schema and some on the old. We need a clear before/after boundary.
- A migration that runs from inside the application process on startup couples the migration step to the app's own crash-and-restart loop. A bad migration becomes a crash loop, not a deploy failure with a clean rollback.
- Local development needs a fast path that does not require generating and committing a migration for every speculative schema change. Senior devs prototype against `db:push`; PRs land with `db:generate` + a reviewed SQL file.

## Decision

**Production: pre-deploy migration step, separate from the application process.**

- Each deploy runs `pnpm --filter @vellum/api db:migrate` as a release task before any new application instance starts serving traffic. On Railway this is `release_command` in the service config; on Fly.io it is `[deploy] release_command` in `fly.toml`; on Kubernetes it would be an init container or a one-shot Job.
- The migration command runs against the production `DATABASE_URL`. It is idempotent: re-running it after success is a no-op because Drizzle tracks applied migrations in `__drizzle_migrations`.
- If the migration step fails, the deploy aborts and the old application instances keep serving traffic on the old schema. No partial-rollout state.
- The application process itself never runs migrations. `apps/api`'s `main.ts` does not call `migrate()`. If it ever needs to, that is a separate ADR.

**Rollback: forward-only, no down migrations.**

- We do not write or run `down.sql`. Rolling back a bad schema change means writing a new forward migration that undoes the damage and deploying it the same way.
- The reasoning: down migrations are written when the schema is wrong but not yet in production. The moment data is written under the new schema, a mechanical down migration loses that data or leaves it orphaned. A forward fix-up makes the recovery deliberate and reviewable.
- For the rare case where a migration breaks production and a forward fix is too slow, the recovery path is restore-from-backup, not run-the-down-migration.

**Local development: hybrid.**

- During feature work on a branch: `pnpm --filter @vellum/api db:push` writes the current schema directly to a local Postgres without producing a migration file. Fast iteration, no churn on the migrations directory.
- Before opening a PR: `pnpm --filter @vellum/api db:generate` produces a reviewed SQL migration. The PR contains the SQL file as part of the diff; reviewers read it. `db:push` artifacts must not leak into PRs.
- A pre-commit check (added in the migration setup PR) blocks committing a schema change without a corresponding migration file.

## Options considered

**In-process migration on startup.** Simple, common in tutorials. Rejected for the reasons above: couples migration to the crash-and-restart loop, makes failed migrations look like app crashes, and creates split-schema states across rolling deploys.

**Idempotent SQL run by ops on a manual schedule.** Honest and explicit but slow. Adds human latency to every deploy and invites drift between what is in the migrations directory and what is in production.

**Drizzle `migrate()` API called from a one-off script.** This is what `db:migrate` script does. The decision is _where_ to call it from; the API itself is right.

**Allow `down.sql` for symmetry.** Considered. Rejected because we have not seen a case yet where a mechanical down migration would actually be safe. The discipline of forward-only forces every recovery to be a deliberate, reviewed change.

## Consequences

The deploy pipeline now has a migration step that can fail and abort the deploy. Operators see migration failures as deploy errors, not as application errors, which is what we want. On the first day a deploy aborts due to a bad migration, the team writes the forward fix and re-deploys; no manual rollback dance.

`apps/api` boots faster because it does not contact the migrations table on every cold start. The trade is that a developer who forgets to run migrations locally will see runtime errors against a stale schema. The hybrid local workflow (`db:push` for iteration) keeps this friction low.

Forward-only migrations have a learning cost. The first time someone wants to "undo" a migration they will reach for `drizzle-kit drop` or write a manual down SQL. The PR review needs to catch that and steer toward a forward fix. The README in `apps/api/src/db/` documents this in plain text so it is the first thing a contributor reads.

Backup posture becomes load-bearing. Forward-only recovery from data corruption depends on point-in-time recovery being available. Whichever Postgres host we end up on (Neon, Supabase, RDS) needs PITR enabled and tested; otherwise the rollback story does not actually exist.

`db:push` artifacts must not leak into PRs. A pre-commit check, plus reviewer discipline, enforces this. Without it the migrations directory drifts and the production migration step starts to lie about the schema.
